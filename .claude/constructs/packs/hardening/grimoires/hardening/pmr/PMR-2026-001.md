## Root Cause

### Classification

| Dimension | Value |
|-----------|-------|
| **Category** | `data-format-mismatch` |
| **Subcategory** | Compound entity ID format incompatible with BigInt parsing |
| **Trigger** | Migration from subsquid to Envio indexer (`26ee6fa3`, Feb 14) |
| **Contributing Factors** | `type-coercion-failure` (BigInt parse failure on non-numeric string), `stale-assumption` (code comments and logic assumed old data format) |

### Root Cause Taxonomy

Categories (pick one primary, zero or more contributing):

- `data-format-mismatch` — ID format, timestamp unit, encoding difference
- `type-coercion-failure` — implicit cast, BigInt parse, Number overflow
- `missing-validation` — no input check, no schema validation, no assertion
- `stale-assumption` — code comment, test fixture, or logic assumes old behavior
- `boundary-crossing` — data changes meaning when crossing API/frontend/contract boundary
- `infrastructure-dependency` — RPC expiry, indexer lag, service unavailability
- `missing-error-handling` — unhandled exception, silent failure, swallowed error
- `configuration-drift` — env var, chain ID, endpoint URL mismatch

### Narrative

This incident exhibits a **two-phase failure pattern**:

**Phase 1 (Subsquid Era, Feb 8–14)**: The subsquid RPC subscription expired around Feb 8, causing the indexer to lag behind the Berachain chain head. Loans became invisible in the UI because the indexer wasn't returning current data. A user (Issue #58) lost 24 NFTs to liquidation because their loans appeared non-existent in the Repay UI. The subsquid RPC was fixed and indexer re-synced on Feb 11, but the infrastructure dependency without monitoring had already caused harm.

**Phase 2 (Envio Era, Feb 14–22)**: The migration to Envio on Feb 14 (`26ee6fa3`) was driven by the urgency to eliminate the subsquid infrastructure dependency (ADR-001: 88% cost reduction, $2,500→$300/mo). The migration correctly updated all 12 API routes to query Envio's entity format, but introduced three regressions in the **consumer layer** — the frontend components that receive API data and pass it to smart contracts or UI rendering.

**Bug 1 — Compound Loan ID (CRITICAL)**: Envio uses compound entity IDs in the format `"80094_backing_157"` (chainId_entityType_numericId). Subsquid used simple numeric IDs like `"157"`. The frontend components passed raw Envio compound IDs directly to `BigInt()` for smart contract calls. `BigInt("80094_backing_157")` throws a `SyntaxError`, causing `payLoanBack()` and `backingLoanExpired()` contract calls to fail silently. Users could see their loans but could not repay them.

**Bug 2 — Timestamp Unit (MEDIUM)**: Envio stores timestamps as Unix seconds (e.g., `1700000000`). Subsquid stored milliseconds. The activity route passed raw Envio timestamps to `new Date(Number(timestamp))` which interprets the value as milliseconds, producing dates in 1970. All activity entries displayed as "56 years ago." A misleading code comment — `"Timestamp is already in milliseconds, no need to multiply by 1000"` — contributed to the bug persisting through review.

**Bug 3 — Missing chainId (HIGH)**: All `usePublicClient()` and `useWalletClient()` calls in backing/loan components relied on the wallet's active chain. If the user was on a different network, read calls would go to the wrong chain and return empty/incorrect data.

The irony: the migration was prompted by the subsquid failure (Phase 1), and the urgency to fix it contributed to insufficient validation of the new data format's downstream effects (Phase 2). ADR-048 explicitly warned about "higher blast radius" from consolidated indexer usage — this prediction materialized.

## Fix Summary

### Commit `48a6ce01` — Loan ID Parsing + Chain Pinning

- Added `getOnChainLoanId()` helper to extract numeric ID from compound Envio format (`"80094_backing_157"` → `"157"`)
- Applied to all `BigInt(loanId)` call sites in `repay-loan.tsx` and `expired-loans-counter.tsx`
- Pinned all `usePublicClient()` calls to `berachain.id` across 7 files
- Also pinned `useWalletClient()` calls (later reverted in next commit due to TypeScript narrowing issue)

### Commit `72a84748` — Timestamp Correction

- Changed `new Date(Number(timestamp))` to `new Date(Number(timestamp) * 1000)` in `Activity.tsx`
- Updated code comment from `"Timestamp is already in milliseconds"` to `"Envio timestamps are in seconds (Unix epoch), convert to milliseconds"`

### Commit `65efc2f9` — TypeScript Fix for WalletClient

- Removed `chainId` from `useWalletClient()` calls (narrows return type to `never` due to wagmi's type inference)
- Kept `chainId` pinning on `usePublicClient()` for read operations
- Affected: `loan-parameter-form.tsx`, `repay-loan.tsx`, `expired-loans-counter.tsx`, `purchase-dialog.tsx`

## Structural Insight

This incident exhibits a **two-phase failure pattern**:

1. **Phase 1 (Subsquid Era, Feb 8–14)**: Subsquid RPC expired → indexer lagged → loans invisible → user forced to default. Root cause: infrastructure dependency without monitoring.

2. **Phase 2 (Envio Era, Feb 14–22)**: Migration to fix Phase 1 introduced new regressions (compound ID format, timestamp units). Root cause: API route tests validated the transform layer but not the consumer layer (frontend → contract).

The irony: the migration was prompted by the subsquid failure (Phase 1), and the urgency to fix it contributed to insufficient validation of the new data format's downstream effects (Phase 2). ADR-048 explicitly warned about "higher blast radius" from consolidated indexer usage — this prediction materialized.

**Key pattern**: When migrating data sources, the API transform layer is necessary but not sufficient. The test boundary must extend to every terminal consumer of the transformed data, especially when those consumers perform type-sensitive operations (like `BigInt()` parsing or `new Date()` construction). The 82-test suite with 100% pass rate validated the wrong layer — the cause (API routes) instead of the effect (frontend components).

**Cross-session learning**: This two-phase pattern — where fixing one incident under urgency introduces another — is a structural vulnerability of time-pressured incident response. The migration checklist protocol (H4) is the systemic defense: it forces consumer-tracing regardless of urgency, converting a one-time lesson into a reusable gate.
